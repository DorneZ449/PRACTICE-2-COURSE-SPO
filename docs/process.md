# Этапы и роли — практическая работа «Генератор паролей»

> Учебная практика, оценка индивидуальная. Проект общий, выполнен
> командой из двух участников. В рамках своей части каждый участник
> выступает в нескольких ролях: аналитика, дизайнера, frontend-
> и backend-разработчика.

## Участники и распределение работ

| Участник | Основной вклад |
|----------|----------------|
| **Скуратов Дмитрий** | Структура проекта, FastAPI base, оценка надёжности, HTML-шаблоны и стили, адаптивность, тесты и документация. |
| **Скуратов Алексей** | Криптостойкая генерация паролей, SQLite-история, клиентский JavaScript, страница истории, Figma-макет и prototype-связи. |

## 1. Проектирование (Аналитик)

### Функциональные требования

| ID  | Требование |
|-----|-----------|
| F-1 | Пользователь вводит **длину пароля**. В UI слайдер ограничен диапазоном 6–32 (по ТЗ), REST API принимает 6–64. |
| F-2 | Пользователь выбирает классы символов: **A–Z, a–z, 0–9, специальные**. |
| F-3 | Должен быть выбран **минимум один** класс — иначе ошибка. |
| F-4 | Пароль генерируется **криптографически стойко** (`secrets`). |
| F-5 | Программа **оценивает надёжность** пароля и показывает её визуально. |
| F-6 | Поддерживается **копирование пароля в буфер обмена**. |
| F-7 | Все сгенерированные пароли **сохраняются в SQLite** и доступны на странице «История». |
| F-8 | История поддерживает **очистку** и **удаление одной записи**. |
| F-9 | Доступен публичный **REST API** с тем же функционалом, что и UI. |

### Нефункциональные требования

| ID    | Требование |
|-------|-----------|
| NF-1  | Адаптивный UI: Desktop ≥ 1280 px, Tablet ≥ 768 px, Mobile ≥ 360 px. |
| NF-2  | Тёмная тема, современный «glass-morphism» стиль (по Figma-макету). |
| NF-3  | Web-приложение на **FastAPI** + **SQLite**. |
| NF-4  | Не использовать сторонние ключи/секреты, всё работает локально. |
| NF-5  | Покрытие тестами ключевой логики (генератор, надёжность, REST). |
| NF-6  | Локальная SQLite-база и открытый REST предназначены только для учебного запуска на одной машине; это фиксируется в README как security assumption. |

### Входные / выходные данные

- **Вход**: длина пароля (целое 6–64), 4 булевых флага классов символов.
- **Выход**: строка пароля + объект надёжности (`label`, `score`, `percent`, `entropy_bits`, `description`).

## 3. Разработка

### Frontend-разработчик

- **HTML-шаблоны** (Jinja2): `templates/base.html`, `templates/index.html`, `templates/history.html`, `templates/_lock.html`.
- **CSS** в `static/css/style.css` — токены и компоненты повторяют Figma 1:1; адаптивные брейкпоинты 720 / 420 px.
- **JS**:
  - `static/js/app.js` — слайдер длины, чекбоксы, генерация через REST, тост, copy-to-clipboard, горячая клавиша `Ctrl/Cmd+Enter`.
  - `static/js/history.js` — копирование/удаление/очистка истории через REST.

### Backend-разработчик

- **FastAPI** (`app/main.py`): SSR-страницы и REST-эндпоинты, lifespan-инициализация БД.
- **Генератор** (`app/generator.py`): `generate_password()` на `secrets.SystemRandom`, валидация параметров.
- **Надёжность** (`app/strength.py`): шкала Weak/Medium/Strong/Very Strong + расчёт энтропии.
- **БД** (`app/db.py`): SQLite, таблица `password_history`, миграция `entropy_bits`.
- **Pydantic** (`app/schemas.py`): запросы и ответы REST.

### Интеграция

- HTML/CSS подключены через `Jinja2Templates` и `StaticFiles`.
- Frontend вызывает REST: `POST /api/generate`, `POST /api/check`, `DELETE /api/history`.
- Сохранение пароля — атомарно в SQLite в той же транзакции, что и ответ API.

## 4. Тестирование

- **pytest** (`tests/`):
  - `test_generator.py` — границы длины (6/12/24/32/64), наличие всех выбранных классов, ошибка при отсутствии классов, уникальность 50 паролей.
  - `test_strength.py` — корректные лейблы для типовых паролей, монотонный рост энтропии.
  - `test_api.py` — health-check, генерация по умолчанию, кастомные опции, валидация Pydantic, сценарий истории (создать → прочитать → удалить → очистить), рендер HTML-страниц.
- **Ручное тестирование** на Desktop / Mobile (Chrome DevTools, эмуляция Pixel 7, iPhone 14): копирование, тост, история, переходы, REST через Swagger UI.
- Найденные при ревью ошибки фиксируются как pull-request'ы.
